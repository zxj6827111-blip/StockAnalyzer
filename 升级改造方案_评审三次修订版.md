# StockAnalyzer 升级改造方案（评审三次修订版）

## 0. 修订定位

本版本用于吸收前两轮评审意见以及最新一轮从代码实现视角提出的 8 个结构性问题，目标不是再“补几个字段”，而是把方案修正为一份真正可实施、可测试、可验收的学习闭环蓝图。

本次新增补强重点如下：

- 新增 `Feature Schema Registry`，解决特征列漂移与跨版本样本不可直接复训的问题。
- 明确训练器必须新增 `train_on_sample_store()` / `train_on_dataset_manifest()` 入口，保证样本仓能够被直接消费。
- 新增完整测试策略，覆盖 schema、状态机、DuckDB、回填、训练、shadow、兼容性。
- 将 `shadow v2` 定义为“并行重建”而不是“轻量增强”，修正工作量预期。
- 修正 `M3` 仍为 `vector_dim=5` 的结构限制，引入向量配置档案与版本锁定。
- 新增 `Label Policy Registry`，避免标签参数变化导致同一 snapshot 在不同时间生成不同标签。
- 显式区分“当时观测到的特征”与“今天重算出来的特征”，处理回填时间穿越问题。
- 将验收从定性描述升级为量化 KPI。
- 同时保留并收口前两轮已确认的约束：DuckDB 单写者 / WAL、orchestrator hook 接入、blocked 状态规则、监控端点统一注册、向后兼容配置。

一句话总结：

> 这次改造的核心，不再只是“建样本仓”，而是“建一套特征锁定、标签锁定、训练可直连、回填可分级、shadow 可验证、升级可审计的学习协议体系”。

## 1. 基于代码的再确认

最新一轮意见和当前代码是对得上的，且不是抽象风险，而是已经存在的结构缺口。

### 1.1 特征版本锁定缺失是真问题

- `src/stock_analyzer/feature/engineer.py` 的 `FeatureEngineer.transform()` 当前直接产出大规模特征列。
- 这些列名、列顺序、默认填充值策略都依赖代码实现。
- 现有方案虽然提到 `feature_schema_hash`，但没有定义一个正式的 `Feature Schema Registry` 去注册“哪一版特征、列序是什么、缺失值怎么处理、是否允许投影兼容”。

结论：

- 如果没有特征注册表，样本仓最多只能成为审计层，难以稳定支撑跨版本训练。

### 1.2 训练器当前无法直接消费样本仓

- `src/stock_analyzer/models/trainer.py` 当前主路径仍然是：
  `train_on_bars() -> FeatureEngineer.transform() -> build_soup_labels() -> train_on_feature_label()`
- 这说明训练器默认输入仍是 `bars`，不是 `(feature_vector, label)` 样本对。
- 如果只建设样本仓，不同步建设训练器新入口，样本仓就只能辅助追踪，不能形成学习闭环。

结论：

- 最新评审中提到的 #2 是当前最关键的结构性问题，我认同这一判断。

### 1.3 Shadow v2 与 M3 当前实现确实偏轻量

- `src/stock_analyzer/evolution/modules/shadow_online_model.py` 目前仍是轻量在线 stub learner。
- `src/stock_analyzer/evolution/orchestrator.py` 当前初始化 `PatternMemoryStore` 时使用 `vector_dim=5`。
- 同文件 `_m3_vectors()` 也只产出 5 维 `open/high/low/close/log(volume)` 风格向量。

结论：

- 这支持最新评审的两个判断：
  `shadow v2` 不是增强，而是重建；
  `M3` 现状无法承载 outcome / regime / execution 记忆。

### 1.4 测试基础存在，但新方案的测试设计尚未写出来

- 当前仓库已有约 146 个 Python 测试文件。
- 已覆盖 `FeatureEngineer`、`ModelTrainer`、`PatternMemoryStore`、`shadow_online_model`、`OffhoursEvolutionOrchestrator` 等基础模块。
- 但现有方案文档没有给新子系统列出测试边界、回归门槛和验收口径。

结论：

- 不是“仓库没有测试”，而是“新架构没有自己的测试方案”。这必须补齐。

## 2. 总体判断

我对最新这 8 条意见的总体判断是：

- 8 条里大部分都应该直接吸收。
- 其中 #1、#2、#3、#7 应提升为 P0 实施约束。
- 其中 #4、#5、#6 是架构与排期修正项，不处理会导致中后期返工。
- #8 必须转化为量化门槛，否则后面很容易出现“看起来完成，实际没有形成学习能力”的假闭环。

因此，这一版方案做两个关键调整：

1. 把 Phase 1 的目标从“先把样本存起来”改成“先把样本协议建完整，并让训练器能直接消费它”。
2. 把回填与复训从“能补历史数据”改成“要清楚标明样本真实性等级，避免今天的重算污染昨天的真相”。

## 3. 修订后的核心原则

后续设计统一遵循以下原则：

1. 观测真相优先于事后重算。
2. 样本协议优先于模型花样。
3. 训练入口必须以样本仓为一等公民。
4. 特征版本、标签版本、数据集版本必须同时锁定。
5. `shadow v2` 必须并行引入，不直接替换现有生产路径。
6. 大文件高风险模块优先 hook/callback 接入，不做大面积侵入式改写。
7. 每个新存储层都必须自带测试、监控和降级策略。

## 4. 修订后的目标架构

### 4.1 六层结构仍成立，但 P0 合同要补齐

保留此前六层分层：

1. 数据层
2. 样本真相层
3. 训练与候选层
4. 注册与验证层
5. 治理与发布层
6. 交互与产品层

但在“样本真相层”和“训练与候选层”之间，必须新增三套协议：

- `Feature Schema Registry`
- `Label Policy Registry`
- `Dataset Manifest`

只有这三套协议同时存在，样本仓才不是普通日志仓，而是可复训、可比较、可回滚的数据合同层。

## 5. 关键设计修正

### 5.1 样本仓不再只是存储层，而是训练真相层

上一版方案对样本仓的定义还偏“统一沉淀”。这一版修正为：

- 样本仓必须携带可训练合同。
- 样本仓中的样本必须知道自己使用哪版特征与标签。
- 样本仓必须能直接被训练器消费。

否则会出现“建了精美仓库但训练器进不去”的情况。

### 5.2 回填必须处理时间穿越

这里必须做一个原则性区分：

- `observed feature`
  是系统在当时运行时真正看到并写下来的特征
- `recomputed feature`
  是今天用当前代码对历史 bars 再算一遍得到的特征

两者不能等价。

因此建议：

- 训练集必须可以按 `backfill_fidelity_tier` 过滤。
- `gold/silver` 可用于 champion/challenger 训练。
- `bronze` 主要用于研究、探索、冷启动扩样，不直接用于高等级晋升。

### 5.3 Shadow v2 要重新排期

这部分不应再写成“升级 shadow learner”，而应拆成两个阶段：

- `shadow v2 data path`
  把样本仓、manifest、feature/label lock、评估框架先打通
- `shadow v2 learner`
  再建设新的 learner、逐样本对比、门槛指标

原因很明确：

- 当前 `shadow_online_model.py` 是轻量 stub。
- 目标中的 `shadow v2` 需要消费更复杂样本、更多特征、更多治理元数据。
- 这本质上是并行重建而不是小修小补。

### 5.4 DuckDB 并发策略需要写死

延续上一轮意见，继续明确：

- 学习仓、注册表、事件账本都建议使用 DuckDB。
- 但写入模型必须采用单写者策略。
- 建议显式声明：
  `WAL 模式 + 单连接写入服务 + 读写职责分离 + append-first`

建议新增：

- `runtime_ops_service` 统一承接这些写入与状态查询
- pipeline / reconcile / week7 / backfill 通过 service API 或 hook 提交事件，不直接到处开连接写表

### 5.5 Orchestrator 集成必须走 hook/callback

`src/stock_analyzer/evolution/orchestrator.py` 体量已经很大，直接侵入式改造风险高。

建议方式：

- 在 orchestrator 中只加轻量 hook 点
- 新学习子系统通过 callback/service 注入
- 避免把样本仓、训练、registry、shadow v2 的全部逻辑塞回 orchestrator 主文件

## 6. 分阶段实施计划

本轮排期比上一版更保守，也更符合当前代码现实。

### Phase 1a：样本协议骨架

周期：

- 1 到 1.5 周

目标：

- 建立 `Feature Schema Registry`
- 建立 `Label Policy Registry`
- 建立统一 `Signal Snapshot` 基础表
- 通过 hook 在 pipeline 侧写入 snapshot

任务：

- 新建 `learning` 模块基础目录
- 落地 `schema_version`、`code_version`、`data_quality_score`、`sample_weight`
- 给 snapshot 绑定 `feature_schema_id`、`label_policy_id`
- 增加 `feature_capture_mode` 字段
- 在训练配置中增加 `legacy_training_path_fallback_enabled`

验收：

- 任一新信号都能写入 snapshot
- 任一 snapshot 都能追溯到 feature schema 与 label policy

### Phase 1b：Dataset Manifest 与训练器新入口

周期：

- 1 到 1.5 周

目标：

- 建立 `Dataset Manifest`
- 打通 `train_on_sample_store()` / `train_on_dataset_manifest()`
- 让训练器从样本仓直接训练

任务：

- 新建 manifest 表与选择规则
- 新增 trainer 新入口
- 训练 artifact 写入 `dataset_manifest_id`、`feature_schema_id`、`label_policy_id`
- 保留 `train_on_bars()` 作为 bootstrap fallback

验收：

- 不重算特征、不重算标签，也能完成一次训练
- 一次训练可以完整回溯到 manifest

### Phase 1c：历史回填与真实性分级

周期：

- 1 到 2 周

目标：

- 建立 `bootstrap_backfill`、`incremental_backfill`、`repair_backfill`
- 处理 `pending -> label_matured -> reconciled -> fully_matured`
- 为历史样本打上 `gold/silver/bronze` 忠实度等级

任务：

- 从 warehouse、reconcile、week7、suggestions 历史中回填
- 区分 observed 与 recomputed feature
- 落地 `backfill_fidelity_tier` 与 `backfill_source`
- 生成首批可复训 manifest

验收：

- 可以导出仅 `gold/silver` 的训练集
- 任一样本都能说明自己是“真实观测”还是“事后重算”

### Phase 2a：模型注册表与 shadow 数据链路

周期：

- 1 周

目标：

- 建立 model registry
- 打通 `shadow dataset builder`
- 让 shadow v2 可以读取样本仓与 manifest

任务：

- 新建 registry 表
- 落地 role / lifecycle 双状态
- 补全 blocked 触发与退出规则
- 构建 champion vs shadow 逐样本对比基础报告

验收：

- 新模型可注册、可绑定 manifest、可进入 shadow 链路
- 不影响现有生产 champion

### Phase 2b：Shadow v2 并行重建

周期：

- 1.5 到 2 周

目标：

- 建立真正可用的 `shadow_online_model_v2`
- 引入 richer feature、execution feature、event feature

任务：

- 新建 `shadow_online_model_v2.py`
- 引入单独状态存储与指标存储
- 构建逐样本收益、校准、执行差异报告
- 保留旧版 shadow 并行运行，直到 v2 稳定

验收：

- shadow v2 可以在一段窗口内稳定运行
- 与旧版 shadow 并行时不干扰主链路

### Phase 3：执行感知学习与晋升门

周期：

- 1 到 2 周

目标：

- 建立 execution-aware reranker
- 把 execution drift、reconcile drift 纳入 gate

前置门槛：

- `fully_matured` 样本达到最低数量门槛
- 每个关键 bucket 的样本量不低于最低阈值

任务：

- 构建执行风险标签
- 训练执行风险模型
- 在晋升门中读取 execution-aware 指标

验收：

- promotion 不再只看 AUC/Brier
- execution 指标恶化时可阻止晋升

### Phase 4：M3/M7 与学习反哺

周期：

- 1 到 1.5 周

目标：

- 升级 `M3` 到 profile 化向量记忆
- 升级 `M7` 到事件账本
- 让 M1/M3/M7 可正式进入训练反馈

任务：

- 引入 `M3 Vector Profile Registry`
- 提升 `vector_dim`
- 给 M3 增加 outcome / regime / execution metadata
- 给 M7 增加事件账本、去重、TTL、归档、有效性追踪

## 10. 量化验收 KPI

这一版开始，验收必须尽量量化。

### 10.1 样本闭环 KPI

- 新产生的有效信号中，`snapshot` 写入成功率 `>= 99%`
- 可成熟样本中，`maturity_status` 正确推进率 `>= 99%`
- 每个进入训练的样本都必须绑定 `feature_schema_id`、`label_policy_id`、`dataset_manifest_id`
- 单个 manifest 中隐式混入多 feature schema 的比例必须为 `0`

### 10.2 训练可复现 KPI

- 同一 `dataset_manifest_id` 重复训练时，样本集合完全一致
- 在确定性模式下，同一 manifest 重跑得到的核心指标漂移不超过预设容差
- 训练 artifact 绑定 manifest / schema / label policy 的覆盖率为 `100%`

### 10.3 回填真实性 KPI

- `gold/silver/bronze` 分类覆盖率为 `100%`
- `bronze` 样本误入 champion/challenger 训练集的次数为 `0`
- 回填任务不得覆盖已存在的 `observed_snapshot` 原始特征

### 10.4 Shadow v2 KPI

- shadow v2 连续稳定运行窗口不少于 `20` 个交易日或等效样本窗
- 在该窗口内不得影响 production champion 的读写路径
- 至少一项核心指标达到显著改善候选门槛后，才允许进入 challenger 审查

### 10.5 治理与安全 KPI

- 非法状态流转次数为 `0`
- 未绑定 manifest 的模型进入 `approved` 的次数为 `0`
- 样本仓或 registry 故障时，误晋升次数为 `0`

### 10.6 测试 KPI

- 每个新增 P0 模块必须同时具备单元测试和集成测试
- Phase 1 结束时，新增关键路径测试通过率为 `100%`

## 12. 最终建议

最新一轮评审的价值很高，尤其是它把问题从“架构方向”推进到了“训练器到底能不能吃样本仓、历史回填到底算不算真相”这个层面。

我对这轮意见的最终结论是：

- 应当采纳，而且要把其中几条前移成 Phase 1 的硬约束。
- 其中最关键的一条，确实是训练器新入口问题。
- 如果不在 Phase 1 同步解决 `Feature Schema Registry + Label Policy Registry + Dataset Manifest + train_on_sample_store()` 这四件事，整套改造最终会更像“增强版审计系统”，而不是“真正能越来越聪明的学习系统”。

这版修订后的实施顺序建议如下：

1. 先建样本协议，不先卷模型。
2. 先让训练器吃样本仓，不先扩更多特征。
3. 先分清 observed 与 recomputed，不先追求大规模历史回填。
4. 先做 shadow v2 的数据链路，再做 shadow v2 learner。
5. 最后再把 M3/M7 真正并入学习闭环。

如果按这条线推进，系统最终会从：

> “具备演化治理外壳的量化系统”

逐步演进为：

> “有样本真相层、有训练合同层、有影子验证层、有安全晋升层的受控自学习系统”
