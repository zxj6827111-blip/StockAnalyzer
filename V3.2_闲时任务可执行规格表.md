# V3.2 闲时任务可执行规格表

> **定位**：本文档是 V2.0 全景规划的"可交付给开发"版本。不新增模块、不改架构，只将规划中的每个任务变成可执行的工程规格。
> **适用对象**：Idle Queue 执行引擎开发者、DAG 调度器维护者。
> **版本**：V3.2 · 2026-03-01 · 基于 V2.0 全景规划 + GPT5.2/CODEX5.3/GROK 六轮交叉评审终稿

## 〇、统一约束（所有闲时任务强制继承）

### 0.1 Staging 落盘根路径与逻辑日期限制

```yaml
trade_date: "最近一个收盘完成的交易日（T）"
run_date: "任务启动当日自然日"
logical_trade_date: "所有任务归属以 trade_date 为准。周五晚至周日晚的所有任务，归属至同一个 trade_date"
trade_date_freeze: "trade_date 在窗口启动时计算一次并冻结，跨午夜不变。直到 effective_hard_stop + 30s 完成后才允许刷新"
idle_outputs_root: "staging/idle_cache/{trade_date}/{task_id}/"
# 示例: staging/idle_cache/20260301/WD-P0-01/

# 目录规范（强制统一）：
# 最终输出目录 = idle_outputs_root + {task_output_subdir}
# 示例: staging/idle_cache/20260301/WD-P0-01/data_quality/report.json

cache_key_format: "{snapshot_id}_{task_version}_{random_seed_if_any}"

# 禁止写入位置（硬编码黑名单）：
forbidden_write_paths:
  - "data/"           # 生产数据目录
  - "config/"         # 生产配置目录
  - "suggestions/"    # 生产提案目录（由 V1.6 门禁专用）
  - "artifacts/"      # 生产制品目录（全局禁写）

# 白名单豁免（仅限以下任务 + 子路径 + 操作类型）：
# ❗ 执行优先级：write_whitelist 优先于 forbidden_write_paths。
#    引擎必须先查白名单（匹配 task + realpath(path) + action 三元组），命中则放行；
#    未命中才查黑名单，命中则 fail-fast。
write_whitelist:
  - task: "WE-P2-08"
    paths: ["artifacts/faiss_snapshots/", "artifacts/shadow_logs/"]
    actions: ["compress", "delete_via_queue"]  # 仅允许压缩归档和标记删除

# 白名单合并策略（全局 + 任务级）：
whitelist_merge_policy:
  merge_strategy: "union(global.write_whitelist, task_manifest.write_whitelist)"
  evaluation_order: "先检查合并后的 effective_write_whitelist，再检查 forbidden_write_paths"
  on_conflict: "若 task/path/action 任一不匹配则视为未命中白名单，继续黑名单判定"

# 防逃逸校验：
# 所有输出路径必须 realpath() 后再做前缀校验；发现软链接或越界（../）立即 fail-fast，并记入违规报告。
```

> 所有闲时任务的输出（缓存、报告、checkpoint、诊断日志）**强制写入** `idle_outputs_root`，不得直接写入生产目录（白名单豁免除外）。`precompute/` 路径作为 `idle_outputs_root` 下的子目录存在。

### 0.2 Checkpoint 统一约定

```yaml
checkpoint_policy:
  must_checkpoint_every_minutes: 10       # 长任务每 10 min 必须写一次 checkpoint
  checkpoint_trigger_mode: "time_or_milestone"  # 时间阈值或任务里程碑，先到先写
  checkpoint_format: "{task_id}_ckpt_{timestamp}.json"
  checkpoint_location: "staging/idle_cache/{trade_date}/{task_id}/checkpoints/"
  max_checkpoint_retention: 10            # 保留最近 10 个 checkpoint，便于长回测任务恢复
```

### 0.3 硬截止时序（工作日，对齐 V1.6 TimeGuard + 盘前保护）

```yaml
time_alignment_policy:
  premarket_time_source: "config.scheduler.premarket_time"
  base_hard_stop: "08:45"
  effective_hard_stop: "min(base_hard_stop, premarket_time)"
  effective_soft_stop: "effective_hard_stop - 10min"
  effective_report_deadline: "effective_hard_stop - 7min"
  hard_kill_grace_seconds: 30
  precedence: "取更早者，避免与盘前任务抢资源"
  time_guard_sync_required: true
  time_guard_sync_check: "assert(TimeGuard.open_auction_guard.start == effective_hard_stop)"
  on_sync_mismatch: "fail_fast_disable_idle_queue_and_raise_alert"
# 例：premarket_time=08:30 时，
# effective_soft_stop=08:20, effective_report_deadline=08:23, effective_hard_stop=08:30
```

```
effective_soft_stop       ── 禁止新开长循环任务，强制各任务写 final checkpoint（未完成则写 partial）后仅允许收尾 + 写报告
effective_report_deadline ── 晨报必须输出完成
effective_hard_stop       ── SIGTERM 发送给所有仍在运行的闲时进程
effective_hard_stop+30s   ── SIGKILL 强制终止（30s 宽限期结束）
```

### 0.4 资源抢占（High/Low Watermark 双重防抖与协作让路）

```yaml
resource_yield_thresholds:
  workday:
    cpu_high_watermark: 70    # 超过 70% → 全面对实盘让路
    cpu_low_watermark: 50     # 回落至 50% 及以下才唤醒
    disk_io_high_mbps: 100    # 磁盘 I/O 超过 100 MB/s → 让路
    disk_io_low_mbps: 50      # 磁盘 I/O 回落至 50 MB/s 及以下才恢复
  weekend:
    cpu_high_watermark: 85
    cpu_low_watermark: 60
    disk_io_high_mbps: 150
    disk_io_low_mbps: 80
  yield_action: 
    action: "协作式让路（引擎不发 OS 信号，直接设置全局 pause_flag；各任务每 5 秒检查此 flag 并于最近的 checkpoint 后主动 sleep）"
    pause_flag_check_interval_seconds: 5  # 任务循环检查 pause_flag 的间隔
    min_pause_seconds: 300    # 强制冷却至少 5 分钟，防止 pause/resume 反复震荡颠簸
    resume_action: "清除 pause_flag（恢复）"
  live_trading_override: "盘中若检测到实盘紧急任务，立即设置全局 pause_flag，所有闲置任务必须让出资源，无视阈值"
```

### 0.5 统一降级与告警升级 TTL（按任务频率分层）

```yaml
stale_data_policy:
  ttl_count_unit: "scheduled_runs"               # 以计划运行次数计数，不按自然日
  ttl_counter_rule: "仅在任务本应运行且最终走 fallback 时计数；未到运行窗口不计数"
  # 按任务调度频率分层 TTL，防止周末低频任务误触告警
  workday_tasks:                             # WD-* 系列（每日运行）
    stale_ttl_runs: 2
  weekend_tasks:                             # WE-P0/P1 系列（每周运行）
    stale_ttl_runs: 9                        # 7次计划运行 + 2次容错
  low_frequency_tasks:                       # WE-P1-07 灾备（每两周运行）
    stale_ttl_runs: 16                       # 14次计划运行 + 2次容错
  stale_escalation_policy: "连续 fallback ≥ 对应 stale_ttl_runs → 触发 PageDuty 级别高危阻断告警，不再允许 fallback"
  fallback_counter_reset_rule: "任一任务成功运行 1 次即将该任务的 fallback 连续计数清零"
  escalation_recovery_policy:
    manual_ack_required: true
    auto_unblock_after_consecutive_success_runs: 2
    unblock_action: "解除 fallback 阻断并恢复正常调度"

staging_cleanup_retention:
  idle_cache_retention_days_workday: 14     # 日常缓存保留 14 天（按 WD 运行频率折算，覆盖 stale_ttl_runs + 充足余量）
  idle_cache_retention_days_weekend: 14     # 周末缓存保留 14 天（按 WE 运行频率折算，覆盖 stale_ttl_runs + 跨周缓冲）
  idle_cache_retention_days_low_freq: 18    # 低频缓存保留 18 天（按双周运行频率折算，覆盖 stale_ttl_runs + 缓冲）
  # 统一由 WE-P2-08 执行 TTL 清理，且仅限 staging 目录
```

### 0.6 成功标准（验收口径）

## 一、工作日夜间任务规格（周一至周四，20:30 – 次日 effective_hard_stop+30s）

### 🔴 P0 级别 — 防守与体检

---

#### WD-P0-01：数据质量夜检

| 维度 | 规格 |
|------|------|
| **任务 ID** | `WD-P0-01` |
| **优先级** | P0（最高，队列头部） |
| **输入** | 生产 `data/` 目录（**只读**） |
| **输出路径** | `staging/idle_cache/{trade_date}/WD-P0-01/data_quality/report.json` |
| **时间预算** | `max_wall_time = 30min` |
| **CPU 预算** | 单线程，≤ 20% CPU |
| **I/O 预算** | 顺序读，≤ 50 MB/s |
| **Checkpoint** | 每校验完一个表/一批股票写一次进度 |
| **校验项** | 缺失 K 线、复权异常、停牌/ST/涨跌停状态、checksum 对账、行数对账 |
| **失败降级** | 单表校验失败 → 标记 `degraded`，跳过该表继续；全部失败 → 沿用上一日报告 + `⚠️ STALE` 标记 |
| **数据不可用替代** | 无（本任务本身就是数据可用性检查） |

## 四、Idle Queue 引擎开发接入规范

```yaml
# 每个闲时任务注册时应提供的元数据：
idle_task_manifest:
  task_id: "WD-P0-01"                      # 唯一标识
  priority: "P0"                             # P0 > P1 > P2
  schedule: "workday"                        # workday | weekend | both
  phase: 1                                   # 对应 Phase 0-3 落地阶段
  depends_on: []                             # 前置依赖任务列表，如 ["WD-P0-01"]
  gate_conditions: {}                        # 门禁条件，如 {"WD-P0-01.status": ["ok", "degraded"]}
  max_wall_time_minutes: 30                  # 超时后写 partial + 退出
  cpu_limit_percent: 20                      # CPU 硬上限
  io_limit_mbps: 50                          # I/O 硬上限（建议值）
  checkpoint_interval_minutes: 10            # checkpoint 间隔
  output_root: "staging/idle_cache/{trade_date}/{task_id}/"
  task_output_subdir: ""                     # 语义子目录（如 "data_quality/"）；最终输出路径 = output_root + task_output_subdir
  forbidden_write_paths: ["data/", "artifacts/", "config/", "suggestions/"]
  write_whitelist: []                        # 如需豁免，格式: [{task, paths, actions}]
  effective_write_whitelist: []              # 运行态字段；由 union(global.write_whitelist, task_manifest.write_whitelist) 计算得到
  # 周末调度策略字段（仅 schedule=weekend 的任务需填）：
  must_run: false                            # 是否强制运行（即使超时也要挤占后续时间）
  defer_policy: "next_weekend"               # 窗口不足时的延后策略：next_weekend | skip | none
  rotating_priority: 0                       # 轮换优先级（上周未完成 → 本周 +1）
  max_defer_runs: 2                          # 最多允许连续延期次数，达到后强制排队执行
  force_run_on_disk_usage_pct: 70            # 磁盘使用率达到阈值时强制插队运行（即使为 P2）
  retry_policy:
    max_retries: 1                           # 最多重试 1 次
    retry_delay_minutes: 5
    retry_only_on: ["transient_io_error", "network_timeout", "file_handle_busy"]  # 仅对瞬态错误重试
    no_retry_on: ["data_unavailable", "forbidden_path", "schema_mismatch"]        # 确定性错误直接走降级
  degradation:
    on_timeout: "write_partial_report"       # 超时策略
    on_data_unavailable: "skip_with_warning" # 数据不可用策略
    on_full_failure: "use_last_successful"   # 全败策略
    fallback_description: "..."              # 具体降级说明（见上方各任务详情）
```
