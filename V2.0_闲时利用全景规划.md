# V2.0 离线算力调度与进化自治方案 (全时段闲时利用)

> **核心哲学**：“空闲多 = 系统在待命安全状态，而不是忙到冒烟容易出错。宁可让算力闲着，也不让它冒险。”
> **设计目标**：将 V1.6 系统的每周 80~100 小时算力空闲压缩至 40~60 小时，专注于“只读型、诊断型”的高价值低风险任务，不改动任何交易行为。

## 一、 系统工程铁律（不可逾越的红线）

在执行任何闲时任务时，必须严格遵守以下 5 条系统级铁律（融合 CODEX 5.3、GPT 5.2 与 GROK 建议）：

1. **绝对隔离（只读生产）**：闲时任务一律“只读”生产数据，任何文件/数据库写入必须隔离到 staging（预发）或实验目录，绝不污染实盘环境与训练集。
2. **优雅可中断（Graceful Shutdown）**：所有任务必须支持随时中断与检查点（Checkpoint）。一旦 CPU/磁盘 IO 超出安全阈值（如工作日 CPU > 70%，周末 > 85%），或实盘任务需要抢占资源，闲时任务必须立即停机让路。
3. **硬截止安全锁（Hard Deadline）**：工作日早上 **08:30** 为系统硬截止线。08:25 向所有闲时进程发送 SIGTERM 信号，给予 **300 秒优雅退出宽限期**（Grace Period）用于完成当前 checkpoint 写入并安全释放资源；若 08:30 仍未退出，则 SIGKILL 强制终止。严禁无宽限期的直接 SIGKILL，以防 staging checkpoint 写入中断导致残缺快照污染下游数据链路。
4. **失败无缝降级**：若夜间某个诊断或体检任务（如模型监控）失败，系统不崩溃，自动沿用上一交易日的成功结果，并仅作 ⚠️ 警告标记。
5. **决策门禁化**：闲时任务不直接修改任何交易行为、策略参数或持仓。它仅产出“报告”、“诊断归因”或“新提案（Proposal）”，最终上线仍需跨越 V1.6 的六道门禁系统。

---

## 二、 核心架构：闲时优先级队列（Idle Priority Queue）

摒弃硬编码的定时器（Crontab）思维，采用基于现有 DAG 调度器的 **按需闲时队列（Idle Queue）机制**。
在不影响实盘和 V1.6 主线任务的前提下，按 P0 -> P1 -> P2 的优先级依次消耗算力。

* **P0（防守/体检）**：最高优先级，确保系统明天不会因为低级错误崩溃。
* **P1（诊断/回放）**：找出亏钱/赚钱的归因，防止“复盘断链”。
* **P2（进攻/研究）**：在极度空闲时（如周末），进行算法算力强消耗的任务（挖矿、大回测）。

## 三、 工作日夜间闲时排程（周一至周四，20:30 - 次日 08:30）

在 V1.6 盘后数据处理与常规训练结束后，若资源允许，依次抽取队列执行：

### 🔴 P0 级别必跑（防守与体检）
1. **数据质量夜检**：校验缺失 K 线、复权异常、停牌/ST/涨跌停状态，进行 checksum 与行数对账，防 silent corruption。*(收益最大，风险最低)*
1.  **数据质量夜检**：校验缺失 K 线、复权异常、停牌/ST/涨跌停状态，进行 checksum 与行数对账，防 silent corruption。*(收益最大，风险最低)*
2.  **失败样本深挖与误杀收敛**：对近 1-4 周的亏损/踏空/误杀样本做归因，统计规则触发次数与误杀率。*(提升实战胜率的关键)*
3.  **模型特征漂移监控（PSI）**：跑一次 PSI（Population Stability Index）检验，预警市场风格切换导致的特征分布漂移。
4.  **相关性与多因子暴露扫描**：计算当前持仓的行业集中度、Barra 风格因子暴露，防止“一锅端”风险。

### 🟡 P1 级别诊断（计算密集型）
5.  **全市场预演选股**：提前用收盘数据计算好 5000 只个股的静态指标并缓存（缓存路径强制为 `staging/precompute/`，并携带 `available_at=T-1 23:59` 时间戳标记，防止主系统误读为最新 T 日数据引入未来函数），将明日早盘决策时间压缩至几秒。
6.  **蒙特卡洛情景压力测试**：采用"历史分块重采样（Block Bootstrap）+ 分情景"”计算 $VaR_{99}$ 与 $CVaR$，而非纯高斯随机漫步。
7.  **板块轮动与情绪雷达**：轻量计算申万一级行业及热门概念概念的主力资金/动量轮动图。

### 📄 08:20 自动输出（终点）
*   **《自动晨报与作战手册》**：推送数据体检报告、候选股快照、风险暴露、参数漂移告警，供开盘前 10 分钟快速决策。

## 四、 周末全天重负载排程（周六下午至周日深夜）

周末在跑完 V1.6 的 M3（FAISS 重建）与 M8（因子挖掘）等主线任务后（通常在周六中午结束），利用余下长达 20-30 小时的窗口进行长期深度验证：

### 🔴 P0 级别必跑（深层系统体检）
1. **全链路 Soak Test（浸泡稳定性演练）**：用加速模式跑最近 30-60 个交易日的**仅 mock 数据源的逻辑流程**（严禁触发真实 BacktestMatcher 或使用当前系统时间作为 as-of），提前发现缓存泄漏、状态机卡死、磁盘堆积等系统隐形 Bug。
2. **可复现性防断链审计**：随机抽取 10-30 个近期上线的 proposal_artifact 做同数据/同种子的一键重放，校验哈希匹配度，防止“赚在回测，死在实盘”。

### 🟡 P1 级别诊断（调用现有模块，长耗时）
3. **5年全历史深度滚动验证**：复用现有 BacktestMatcher，设置 `window=1260` 交易日。产出 5 年 IR 漂移图与极端市场存活率，反馈给 Score Fusion 体系反脆弱。
4. **影子盘“反事实”深度回放**：拆解如果使用 Challenger（挑战者）策略会怎样的因果，归因到入场/出场/过滤/风控具体环节。
5. **多种子标签优化验证（Ensemble）**：复用现有 M5 动态标签模块，采用 3-5 个不同随机种子重跑，校验标签和预测的稳健性。
6. **成本与滑点敏感性扫描**：使用保守/极保守等多档滑点和手续费参数回放近期策略，测算流动性承载边界。
7. **灾备恢复演练**：在 Staging 环境模拟一次“冷备份恢复全流程”，验证 RTO（恢复时间）和数据一致性。

### 🟢 P2 级别维护（保洁与提速）
8. **存储字典深层保养**：清理超 30 天的冷影子盘日志、压缩 FAISS 旧快照、对 DuckDB/Parquet 进行 Vacuum/Compact 及分区重整，保证 I/O 速度，控制 SSD 容量在健康阈值。

## 五、 分阶段无脑落地路线图（实施策略）

为了保护当前 V1.6 核心代码的开发完整性，杜绝“上下文污染”，V2.0 升级需遵循以下时序推进：

*   **Phase 0（当前阶段）**：**按兵不动。** 专心完成 V1.6 骨架开发，打出稳定的 `v1.6-stable` Git Tag，并在实盘初步观察 2-4 周，收集真实的模块耗时和磁盘 I/O 基线数据。
*   **Phase 1（基础防守接入）**：只需在 DAG Scheduler 加持周末 `if day == Sunday` 分支。优先将 **全历史验证（调用老接口）**、**数据体检对账**、**磁盘冷数据压缩** 上线。
*   **Phase 2（日常诊断接入）**：引入 **工作日 Idle Queue**。加入失败样本深挖、模型漂移监控、可复现性抽查。
*   **Phase 3（全算力进攻接入）**：在系统极度稳定且证明 V2.0 调度框架绝不影响实盘后，逐步开启蒙特卡洛压力测试、Soak Test 等 CPU/IO 重消耗型演练任务。
