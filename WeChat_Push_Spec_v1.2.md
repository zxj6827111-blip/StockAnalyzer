## 1. 当前结论

`v1.0` 和 `v1.1` 都要保留，但职责不同：

- `v1.0` 负责回答“系统理论上应该覆盖哪些推送场景”
- `v1.1` 负责回答“这些消息应该怎么发，才能不打扰人且可执行”
- `v1.2` 负责回答“当前系统上线时，究竟以什么口径执行”

因此，当前推荐口径不是“用 `v1.1` 替换 `v1.0`”，而是：

1. 保留 `v1.0` 的 17 类全景推送矩阵作为范围骨架
2. 引入 `v1.1` 的“事件 + 影响 + 建议动作”模板与状态变化去重
3. 明确 `Dashboard` 是低打扰运行下的配套控制台，而不是替代微信主动告警
4. 以本文件作为当前上线执行版

## 3. 设计原则

### 3.1 一条消息只表达一件事

- 同一条消息不要同时塞入“状态变化 + 复盘摘要 + 多个动作建议”
- 用户应在几秒内看明白“发生了什么”和“现在要不要处理”

### 3.2 告警必须有动作

所有 `P0/P1` 告警，默认采用四段结构：

- 事件：发生了什么
- 影响：当前系统受什么影响
- 建议动作：用户现在该做什么
- 追踪：`symbol / recommendation_id / trace_id / 报告入口`

### 3.3 状态型消息必须边缘触发

以下类型默认只在“进入异常”和“恢复正常”时推送：

- 数据源退化 / 恢复
- 熔断触发 / 解除
- 云冷备离线 / 恢复
- Kill Switch 触发 / 解除

不重复推送“同一异常仍然存在”。

### 3.4 微信负责提醒，Dashboard 负责处理

- 微信适合 `P0/P1` 主动叫醒
- `Dashboard` 适合查看全局状态、详情、对账、确认、手工调整
- 低打扰运行后，更多运行信息应沉淀到 `Dashboard`，而不是全部塞进微信

### 3.5 模板统一，内容中文化

- 标题、正文、动作建议统一使用中文
- 级别、分类、符号、字段顺序尽量固定
- 不再发送偏工程日志式的纯英文通知

## 4. 最终分级与通道模型

| 级别 | 含义 | 默认通道 | 处理原则 |
| :--- | :--- | :--- | :--- |
| `P0` | 必须立即处理 | 企业微信主动推送，必要时主备双发 | 强提醒，不静默，不依赖用户主动打开页面 |
| `P1` | 当天必须知道 | 企业微信主动推送 | 必须带影响和建议动作 |
| `P2` | 日常运行信息 | 企业微信摘要 / 定时消息 / `Dashboard` | 允许聚合，避免高频打扰 |
| `P3` | 参考留档 | `Dashboard` / 周报月报 / 留档 | 默认不打扰，不应占用主通知频次 |

### 6.1 P0：必须立即处理

- 止损 / 卖出指令
- 极端行情熔断触发

- 验收通过报告
- 周报 / 月报
- 遗珠复盘
- 进化提案审批记录

## 7. 标准消息模板

### 7.1 通用结构

```text
[分类] 标题
事件：……
影响：……
建议动作：……
追踪：symbol=... / recommendation_id=... / trace_id=...
```

### 7.2 示例：S级买入信号

```text
[情报] ⚡ S级买入候选
事件：盘中发现 600000 评分 81，达到 S 级阈值
影响：当前允许开仓，候选进入可执行观察名单
建议动作：如无额外主观禁入因素，可按建议仓位 12% 关注
追踪：symbol=600000 / recommendation_id=rec_xxx
```

### 7.3 示例：数据源抓取异常

```text
[运维] ⚙️ 数据源抓取异常
事件：AKShare 主源连续失败 3 次
影响：系统已进入退化模式，暂停新的开仓信号
建议动作：如持续超过 10 分钟，请人工检查数据源与网络
追踪：provider=akshare / trace_id=ops_xxx
```

### 7.4 示例：止损 / 卖出指令

```text
[行动] 🎯 卖出指令
事件：持仓 605888 触发止损阈值
影响：继续持有将超出当前风控许可范围
建议动作：立即减仓或清仓，并在 Dashboard 核对成交与持仓
追踪：symbol=605888 / trace_id=risk_xxx
```

### 7.5 示例：收盘对账提醒

```text
[收盘] 🏦 对账与存活确认
事件：今日收盘后已生成模拟持仓快照
影响：系统运行正常，但需确认券商侧与模拟盘是否一致
建议动作：登录券商 APP 核对；若偏差明显，请执行对账
追踪：trading_date=2026-03-07 / snapshot_id=snap_xxx
```

## 8. Dashboard 协同口径

`Dashboard` 是推送静音后的配套控制台，作用不是替代微信，而是承接更详细、更可交互的能力。

### 8.1 角色定位

- 查看组合、持仓、推荐、风控、缓存、运行状态
- 承接“暂停开仓 / 恢复开仓 / 手工设仓 / 手工平仓 / 快速对账”
- 展示那些不适合高频推送但又需要随时查看的运行细节

### 8.2 当前能力边界

当前仓库已具备以下典型能力：

- `Dashboard` 页面入口：`/dashboard`
- 操作权限开关：`/dashboard/ops/state`、`/dashboard/ops/toggle`
- 快捷指令：`/dashboard/command/quick`
- 快速对账：`/dashboard/reconcile/quick`

### 8.3 使用原则

1. 关键 `P0/P1` 事件必须仍由微信主动推送
2. `Dashboard` 负责查看详情、确认状态、执行低频人工动作
3. 快捷操作默认只允许在模拟盘模式启用
4. 任何需要“立即知道”的风险，不允许仅存在于 `Dashboard`

### 8.4 当前本机验证入口

按当前本机 Docker 映射，已验证可访问入口为：

- `http://127.0.0.1:8001/dashboard`

## 9. 立即执行的 5 项改造

如果本轮只再做一批推送优化，建议按以下顺序落地：

1. 补独立的“策略自毁预警”
2. 补独立的“高风险事件 / 黑名单告警”
3. 补独立的“止损 / 卖出指令”
4. 为数据源、熔断、云冷备统一做状态变化去重
5. 将盘前简报改成“结论优先”文案

## 10. 上线使用建议

1. 以后讨论推送体系，统一以 `WeChat_Push_Spec_v1.2.md` 为当前版本
2. `v1.0` 与 `v1.1` 继续保留，不删除，作为历史设计依据
3. 新功能上线时，优先更新本文件中的“17 类矩阵状态”
4. 若未来切到更强交互模式，可把 `P2/P3` 更多迁移到 `Dashboard`
5. 无论后续如何优化，`P0/P1` 不能因为降噪而失去主动告警能力
